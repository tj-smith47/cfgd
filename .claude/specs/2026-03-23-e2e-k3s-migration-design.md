# E2E Test Migration: KIND → k3s Cluster — Design Spec

**Date:** 2026-03-23
**Status:** Approved
**Depends on:** Windows support (COMPLETE), Linux desktop configurators (IN PROGRESS — no conflict)
**Supersedes:** Docker availability claims in `2026-03-22-e2e-and-workflow-architecture-design.md` — that spec stated "no Docker daemon available" on ARC runners, which was written before the custom runner image was finalized. Docker IS available on ARC runners via the custom image (`registry.jarvispro.io/arc-runner:latest` includes Docker buildx; Docker daemon is provided by the runner infrastructure).

## Goal

Migrate all cfgd E2E tests from KIND (Kubernetes-in-Docker) to the real k3s cluster (10.23.46.201). Tests must run on ARC bare-mode runners (`arc-cfgd`) in CI and locally from the repo. No GitHub-hosted runners for E2E. Every existing test case must pass with same or better coverage.

## Infrastructure Context

- **k3s cluster:** 4 nodes (1 master, 3 workers), k8s v1.33.5+k3s1, accessible at 10.23.46.201:6443
- **ARC runners:** `arc-cfgd` in `actions-runner-system` namespace, custom image `registry.jarvispro.io/arc-runner:latest` with Rust, cargo, sccache, Docker (buildx + daemon), kind, kubectl, helm pre-installed.
- **Container registry:** `registry.jarvispro.io` (ClusterIP 10.43.61.36:5000) in `jarvispro` namespace. k3s nodes can pull from it. Unauthenticated push from within the cluster network.
- **cert-manager:** Installed on the cluster (`cert-manager` namespace). Available for webhook TLS.
- **Network:** Tailscale exposes the subnet via pfSense. This machine and ARC runners both have direct cluster API access.
- **Existing namespace:** `cfgd-system` already exists on the cluster.

## Architecture

### Persistent Infrastructure (always on cluster)

These resources live permanently on the k3s cluster. A pre-flight script diffs them against the current code and applies any changes. They are NOT created/destroyed per test run.

- **CRDs** — generated by `cfgd-gen-crds` (authoritative source; number may grow as CRDs are added)
- **cfgd-operator Deployment** + ServiceAccount + ClusterRole + ClusterRoleBinding
- **Device gateway Deployment** (cfgd-operator with `DEVICE_GATEWAY_ENABLED=true`) + Service + SA + RBAC
- **CSI driver** DaemonSet (via Helm chart with only CSI enabled)
- **Webhook configurations** (ValidatingWebhookConfiguration, MutatingWebhookConfiguration) + Service
- **Webhook TLS** — cert-manager Certificate + Issuer (replaces the self-signed 1-day certs from `generate_webhook_certs`). Cert-manager handles rotation automatically.
- **`cfgd-system` namespace**
- **`cfgd-e2e` ServiceAccount** with scoped ClusterRole for test operations

### Ephemeral Per-Run Resources

Created during test execution, cleaned up after:

- Test CRD instances (MachineConfigs, ConfigPolicies, DriftAlerts, Modules, ClusterConfigPolicies)
- Test namespaces (e.g., `e2e-inject-<run>`, `e2e-team-alpha-<run>`)
- Privileged test pod (for node-level tests)
- OCI artifacts pushed to registry (tagged with run ID)

All ephemeral resources are labeled `cfgd.io/e2e-run=<run-id>` for reliable cleanup. Cleanup uses label selectors (`kubectl delete module -l cfgd.io/e2e-run=<run-id>`), never `--all`, to avoid destroying persistent or other runs' resources.

### RBAC

A dedicated `cfgd-e2e` ServiceAccount in `cfgd-system` with a ClusterRole granting:

- Full access to `cfgd.io` API group (all CRDs + status subresources)
- Namespace create/delete (for test namespaces)
- Pod, Deployment, DaemonSet, Service, Secret, ServiceAccount, ConfigMap CRUD
- RBAC resources (Role, RoleBinding, ClusterRole, ClusterRoleBinding) for test setup
- Webhook configurations (ValidatingWebhookConfiguration, MutatingWebhookConfiguration)
- CSIDriver, CSINode resources (read)
- Events, Leases
- cert-manager Certificate/Issuer resources

A kubeconfig referencing this ServiceAccount (with embedded token) is stored as a Kubernetes Secret (`cfgd-e2e-kubeconfig`) in `actions-runner-system`, mounted into ARC runner pods via a volume/volumeMount in the AutoscalingRunnerSet template. The `KUBECONFIG` env var points to the mounted file. The setup-cluster.sh script includes a `verify-kubeconfig` step that runs `kubectl cluster-info` to confirm connectivity before proceeding.

For local runs, the user's existing kubeconfig works (already has cluster-admin or equivalent).

### Image Pipeline

**Build:** On the ARC runner (or locally), `cargo build --release` all binaries, then `docker build` the 3 images:
- `docker build -f Dockerfile -t registry.jarvispro.io/cfgd:e2e-<sha> .`
- `docker build -f Dockerfile.operator -t registry.jarvispro.io/cfgd-operator:e2e-<sha> .`
- `docker build -f Dockerfile.csi -t registry.jarvispro.io/cfgd-csi:e2e-<sha> .`

**Tag:** `registry.jarvispro.io/<image>:e2e-<short-sha>` — cache-friendly, deterministic. Same commit always produces the same tag.

**Push:** `docker push` to `registry.jarvispro.io`. Idempotent — if the tag exists and matches, push is a no-op. Registry is unauthenticated from within the cluster network.

**Deploy:** Patch the persistent Deployments/DaemonSets to use the new image tag. Kubernetes handles the rollout.

### Concurrency Safety

Each E2E workflow already has `concurrency: group: ${{ github.workflow }}-${{ github.ref }}, cancel-in-progress: true`. This means:

- **Same branch, same workflow:** Only one run at a time. Previous run is cancelled.
- **Same branch, different workflows** (e.g., e2e-node + e2e-operator triggered by same push): Both run `e2e-setup.yml` concurrently. Since image tags are deterministic from git SHA, both produce identical images and identical `kubectl apply` manifests. Concurrent `kubectl apply` of identical manifests is safe (last-write-wins with identical content = convergent).
- **Different branches:** Different SHAs produce different image tags. Concurrent runs would race to set different images on persistent deployments. This is acceptable because: (a) only the `master` branch triggers E2E on push, (b) PR workflows also have per-ref concurrency groups, (c) the "losing" workflow's tests will still work because they wait for the deployment to be ready with their expected image.

### Failure Recovery

If `setup-cluster.sh` fails mid-way (e.g., image push succeeds for cfgd but fails for cfgd-operator):
- The script is idempotent — re-running it completes the remaining steps.
- Persistent infrastructure may be partially updated but not broken (old image still works).
- A `setup-cluster.sh --reset` flag forces full re-apply of all resources regardless of diff status.

## Suite-by-Suite Design

### CLI Suite (`e2e-cli.yml`)

**No Kubernetes dependency.** Tests cfgd CLI against local filesystem fixtures.

**Changes:**
- Drop the `exhaustive-docker` job entirely (Docker-based run exists only for environment isolation; the native run provides the same coverage)
- `exhaustive-native` job: `runs-on: arc-cfgd`, install sops/age, run `run-exhaustive-tests.sh`
- No changes to the test script itself — it doesn't touch Kubernetes (it does source `helpers.sh` for assertion utilities, which still work)

### Node Suite (`e2e-node.yml`)

**Calls reusable `e2e-setup.yml`, then runs 4 test jobs.**

**Privileged test pod** replaces KIND node container:
- Image: `registry.jarvispro.io/cfgd:e2e-<sha>` (has the cfgd binary at `/usr/local/bin/cfgd`)
- Security context: `privileged: true` (required for sysctl writes, kernel module loading)
- No `hostPID` or `hostNetwork` (not needed — tests don't require host PID namespace, and cluster DNS is needed for server tests)
- hostPath mounts:
  - `/host-proc` → `/proc` on host (for sysctl via `/host-proc/sys/...`)
  - `/host-sys` → `/sys` on host (for kernel module state)
  - `/host-etc` → `/etc` on host (for persistence files like `/etc/sysctl.d/99-cfgd.conf`)
- nodeAffinity: require scheduling on a worker node (not the master) — use `kubernetes.io/hostname` in `[k3s-worker-1, k3s-worker-2, k3s-worker-3]`
- Additional packages (procps, kmod, git, curl) installed via image or init container
- Created per test run in the `E2E_NAMESPACE`, deleted during cleanup

**Host side-effect mitigation:** The privileged pod writes to `/etc/sysctl.d/99-cfgd.conf` and `/etc/modules-load.d/cfgd.conf` on the worker node. Test cleanup must remove these files. The drift tests restore sysctl values to their original state. These operations are safe on a dedicated test worker — the k3s worker nodes are not running user workloads sensitive to sysctl changes.

**Helper rewrites in `helpers.sh`:**

| Old (KIND/Docker) | New (k3s/kubectl) |
|---|---|
| `get_kind_node` | `get_test_pod` — returns pod name |
| `exec_on_node cmd` | `exec_in_pod cmd` — `kubectl exec $TEST_POD -n $E2E_NAMESPACE -- cmd` |
| `copy_to_node src dest` | `cp_to_pod src dest` — `kubectl cp src $E2E_NAMESPACE/$TEST_POD:dest` |
| `install_binary_on_node` | Not needed — cfgd is already in the image |
| `install_packages_on_node` | Not needed — image has required packages |
| `start_local_registry` | Not needed — `registry.jarvispro.io` is persistent |

**Binary tests (T01–T11):** `exec_in_pod` replaces `exec_on_node`. cfgd binary is already in the image (no extraction needed). Fixtures copied via `kubectl cp`. Sysctl operations use `/host-proc/sys/` paths (or the pod's own `/proc/sys/` since privileged pods share the host's sysctl namespace).

**Helm tests (T20–T25):** Run directly from the runner (kubectl + helm). No privileged pod needed. Deploy cfgd DaemonSet into a **dedicated ephemeral namespace** (not `cfgd-system` — avoids destroying persistent infrastructure on `helm uninstall`). The Helm values file is replaced with inline `--set` flags or a new ephemeral-specific values file that uses the e2e image tag. The chart's cluster-scoped RBAC resources use run-specific names to avoid collision with persistent infrastructure.

**Server tests (T30–T35):** cfgd checkin from the privileged pod to the persistent device gateway. Gateway URL resolved via cluster DNS: `http://cfgd-server.cfgd-system.svc.cluster.local:8080`. This works because the pod uses cluster networking (no `hostNetwork`), so CoreDNS resolution is available.

**Drift tests (T40–T50):** Sysctl, kernel module, seccomp, certificate, daemon tests all via `exec_in_pod` with the privileged pod.

### Operator Suite (`e2e-operator.yml`)

**Calls reusable `e2e-setup.yml`, then runs operator tests.**

**No privileged pod needed.** All tests are kubectl-based CRD operations.

**Changes:**
- CRDs, operator, webhooks are persistent — pre-flight ensures they're current
- OCI supply chain tests (T18): push to `registry.jarvispro.io` instead of ephemeral local registry. The `cfgd module push` command pushes to the registry using HTTP (insecure) since it's an in-cluster registry. The `OCI_INSECURE_REGISTRIES` env var or `--insecure` flag enables this.
- Test CRD instances created in ephemeral namespace with run-labeled names
- Webhook tests (T12, T15–T17) work against persistent webhook configs, pods created in ephemeral namespace scoped by `namespaceSelector`
- ClusterConfigPolicy tests (T13–T14): use run-suffixed namespace names (e.g., `e2e-team-alpha-<run>`) and label-scoped cleanup
- Cleanup: `kubectl delete module -l cfgd.io/e2e-run=<run-id>` (not `--all`)
- The full setup block at the top of `run-operator-tests.sh` (CRD generation, webhook setup, operator deployment, registry start) is removed — `e2e-setup.yml` handles all of this. The script starts with a "verify infrastructure is ready" check instead.

### Full-Stack Suite (`e2e-full-stack.yml`)

**Calls reusable `e2e-setup.yml`, then runs full-stack tests.**

**Combines all components:**
- Privileged pod for cfgd node operations (checkin, drift)
- kubectl for CRD/operator interactions
- Persistent CSI DaemonSet for volume mounting tests (T11–T12)
- kubectl plugin tests (T13–T15) run from the runner (build cfgd binary, create `kubectl-cfgd` symlink)
- Debug flow (T16) creates pods in ephemeral namespace

**The full-stack script's large setup section (lines 29-98 of current `run-full-stack-tests.sh`) is gutted.** CRD generation, registry setup, webhook TLS, gateway/operator/CSI deployment are all handled by `e2e-setup.yml`. The script replaces this with a compact "verify all components are healthy" check (operator pod running, gateway reachable, CSI DaemonSet ready).

### Reusable Setup Workflow (`e2e-setup.yml`)

**`workflow_call` reusable workflow.** Called by `e2e-node.yml`, `e2e-operator.yml`, `e2e-full-stack.yml`.

Outputs: image tags for cfgd, cfgd-operator, cfgd-csi.

Steps:
1. Checkout + Rust toolchain + sccache
2. `cargo build --release` all binaries (cfgd, cfgd-operator, cfgd-gen-crds, cfgd-csi)
3. `docker build` + `docker push` all 3 images to `registry.jarvispro.io` with `e2e-<sha>` tag
4. Generate CRDs via `cfgd-gen-crds`, diff-and-apply
5. Ensure `cfgd-system` namespace
6. Apply/update operator deployment, gateway deployment, RBAC, cert-manager Certificate/Issuer for webhook TLS, webhook configs, CSI driver — all referencing the new image tag
7. Wait for all deployments/DaemonSets to be ready

This workflow calls `tests/e2e/setup-cluster.sh` which contains the actual logic. The workflow is a thin wrapper that sets up the toolchain and invokes the script.

## Common Helpers (`tests/e2e/common/helpers.sh`)

### Removed

- `KIND_CLUSTER` variable
- `get_kind_node`, `exec_on_node`, `copy_to_node`
- `install_binary_on_node`, `install_packages_on_node`
- `start_local_registry`, `stop_local_registry`, `configure_registry_on_nodes`
- `generate_webhook_certs`, `install_webhook_config` — replaced by persistent cert-manager setup

### Added

- `E2E_NAMESPACE` — `cfgd-e2e-${GITHUB_RUN_ID:-$(date +%s)-$$}` (PID suffix prevents same-second local collision; Kubernetes 63-char DNS label limit is safe — `cfgd-e2e-1234567890` is 20 chars)
- `E2E_RUN_LABEL` — `cfgd.io/e2e-run=${GITHUB_RUN_ID:-local-$$}`
- `REGISTRY` — `registry.jarvispro.io`
- `IMAGE_TAG` — `e2e-$(git rev-parse --short HEAD)`
- `TEST_POD` — name of the privileged test pod (set by `ensure_test_pod`)
- `ensure_test_pod` — deploys privileged pod from template, waits for Running, exports `TEST_POD`
- `exec_in_pod` — `kubectl exec $TEST_POD -n $E2E_NAMESPACE -- "$@"`
- `cp_to_pod src dest` — `kubectl cp src $E2E_NAMESPACE/$TEST_POD:dest`
- `ensure_infrastructure` — diff-and-apply persistent resources. Uses `kubectl diff` to check, `kubectl apply` only if changed. Called by `setup-cluster.sh`, also usable standalone.
- `create_e2e_namespace` — creates ephemeral namespace with run label
- `cleanup_e2e` — deletes ephemeral namespace + cluster-scoped resources by label selector (`-l cfgd.io/e2e-run=<run-id>`)
- `cleanup_host_files` — removes `/etc/sysctl.d/99-cfgd.conf` and `/etc/modules-load.d/cfgd.conf` from host via privileged pod

### Unchanged

All Kubernetes helpers (already kubectl-based):
- `wait_for_pod`, `wait_for_deployment`, `wait_for_daemonset`
- `port_forward`, `wait_for_url`, `wait_for_k8s_field`
- `create_test_module_dir`
- All assertion helpers, test lifecycle helpers

## Workflow Structure

```
e2e-setup.yml          (workflow_call — reusable)
  ↑ called by
e2e-cli.yml            (standalone — no cluster, runs-on: arc-cfgd)
e2e-node.yml           (calls e2e-setup.yml → 4 parallel test jobs)
e2e-operator.yml       (calls e2e-setup.yml → 1 test job)
e2e-full-stack.yml     (calls e2e-setup.yml → 1 test job)
```

All `runs-on: arc-cfgd`. No GitHub-hosted runners for any E2E workflow.

README badges unchanged — workflow file names preserved.

## Local Testing

```bash
# One-time: ensure infrastructure is current (builds images, pushes, applies infra)
bash tests/e2e/setup-cluster.sh

# Run individual suites
bash tests/e2e/cli/scripts/run-exhaustive-tests.sh
bash tests/e2e/node/scripts/run-binary-tests.sh
bash tests/e2e/node/scripts/run-helm-tests.sh
bash tests/e2e/node/scripts/run-server-tests.sh
bash tests/e2e/node/scripts/run-drift-tests.sh
bash tests/e2e/operator/scripts/run-operator-tests.sh
bash tests/e2e/full-stack/scripts/run-full-stack-tests.sh

# Or via Taskfile
task e2e              # all suites
task e2e:cli          # just CLI
task e2e:node         # just node
task e2e:operator     # just operator
task e2e:full-stack   # just full-stack
```

`setup-cluster.sh` builds all binaries (including `cfgd-gen-crds`) before applying CRDs, so no external binary availability is assumed.

Prerequisites: Docker (for image builds), kubectl (configured for k3s cluster), helm, cargo.

## Performance Expectations

| Factor | Before (KIND on ubuntu-latest) | After (k3s on arc-cfgd) |
|--------|-------------------------------|------------------------|
| Cluster creation | ~2-3 min per job (KIND) | 0 (persistent) |
| Image loading | ~1-2 min per job (kind load) | ~30s (docker push, cached layers) |
| Rust build | No cache | sccache on ARC runner |
| Network | Docker networking + port-forwards | Direct cluster network |
| CRD install | From scratch each run | Diff-and-apply (usually no-op) |
| Webhook setup | From scratch each run | Persistent + cert-manager rotation |
| Parallelism | 3 separate workflows | Shared setup + parallel fan-out |
